# Changelog

All notable changes to this project are documented here. Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/); this project will adopt [Semantic Versioning](https://semver.org/) starting at the first release.

## [Unreleased]

### Added

- **Adjustable shower detection controls:** `display_mode: system` now renders
  editable `shower_temperature_rise` and `shower_detection_window` number
  roles inside the Shower Detection banner, using Home Assistant number
  metadata for min/max/step/unit and calling `number.set_value`. Missing
  controls are hidden independently, unavailable controls render disabled, and
  existing shower status/pipe/trigger/re-arm/boost behavior is preserved.
- Added `shower_rearm_temperature_drop` as a third editable shower detection
  control. The active shower banner now calculates "Re-arm at" from trigger
  temperature minus the mapped drop value, while keeping the older 10°C display
  fallback when the entity is not configured.
- **v1.0 polish release controller completion:** the system card now consumes the merged ha-altair-mvhr backend controller entities: `stop_control` exposes Altair Coil 00004 as a normal Off option, stopped units show a clear stopped state and disable airflow particles/fan rotation while preserving live temperatures, airflow calibration has a dedicated panel with availability/start/cancel/progress/status/result/last-calibration fields, and Away/Low/Home/High airflow presets render as editable native number controls. The Airflow gauge now scales measured airflow against the configured maximum (`max_airflow`, then `maximum_airflow`, then `high_airflow`, then profile default, with mapped level only as the final fallback). The cutaway also gets final minor polish to blower clarity, exchanger plate detail, recovery-badge placement, cabinet seams, shadows, and duct collars without changing the responsive layout, temperature colour mapping, gradients, particle system, shower banner, reduced-motion handling, or theme compatibility.
- **Temperature colour tuning:** adjusted the system-overview airflow interpolation so warm tones begin at roughly 18°C. Air in the 15–17°C range now stays low-saturation and only slightly cool, then transitions through warm cream/amber to red by 35°C while preserving the existing independent stream gradients, particles, unavailable fallback, and animation behaviour.
- **System dashboard follow-up:** removed the duplicate internal cabinet label from the system cutaway, moved shower detection out of the header and into a full-width banner below the lower cards, added ready/active/unavailable shower states driven by `shower_detected` plus `boost_active`, surfaced indoor humidity on the Extract air card, made preset airflow controls visible in More controls with an empty state when no backend number entities are configured, added `calibration`/`start_calibration`/`cancel_calibration` aliases, and tightened runtime Off handling/tests so the card only shows Off when a real mode-select option or supported stop-control entity exists.
- **New `display_mode: system`** — a flagship, full-width visual panel aimed at homeowners: a large animated airflow visual (four ducts around a bigger central unit, heat-recovery badge inside it), a compact primary metrics row (airflow, target, recovery, humidity, filter, fans — deliberately no mapped level), simple Mode/Boost controls, and a "More controls" disclosure (collapsed by default) revealing override, mapped level, calibration internals, and individual fan RPM. Two new config options, `show_airflow_animation` and `show_advanced_controls` (both default `true`), are system-mode-only. `display_mode: homeowner` and `display_mode: detailed` are unchanged by this addition. See the README's "System mode" section.
- The airflow visual's duct animation is direction-aware in system mode (Exhaust/Supply flow away from the unit, Extract/Outdoor flow toward it) and only runs when the measured airflow is genuinely positive, the required temperature entities are available, `show_airflow_animation` is on, and `prefers-reduced-motion` isn't set; a slow fan-rotation animation follows the same gating.
- `src/data/availability-summary.ts` / `_dashboardStatus`'s "System OK / Communication issue / Fault detected / Calibrating… / Calibration required" status vocabulary (introduced for the detailed-dashboard rebuild) is now reused by system mode's header and status strip via a new shared `_statusStrip` method — refactored out of `_dashboard` with no change to detailed mode's rendered output.
- 30 new tests in `tests/unit/card-rendering.test.ts` and 3 in `tests/unit/config-schema.test.ts` covering `display_mode: system`: configuration acceptance, the hero visual and its four air paths, the primary-metrics/advanced-diagnostics split (mapped level only in the drawer), mode/boost/override service calls, status logic (communication issue, fault, calibration required, optional-missing-doesn't-warn), Altair's absent bypass row vs. Zehnder's present one, and responsive/reduced-motion CSS.
- **Visual redesign of `display_mode: system`.** The primary-metrics row and chip-style Mode/Boost controls are replaced by a compact header (a `<select>` for mode, a boost status/toggle pill), a larger two-column upper section — an enlarged System Overview visual (bigger unit/fans, directional-arrow icons on all four air paths, a heat-recovery badge in the panel heading, a recoloured blue/orange/grey path palette) beside a new shower-detection panel — and three dedicated lower cards (Airflow, with a semicircular SVG/CSS gauge and no charting library; Temperatures; System Status). The "More controls" disclosure is trimmed to override, calibration internals, and individual fan RPM now that mapped level lives in the Airflow card. Every colour is a Home Assistant theme CSS variable or a `color-mix()` tint against one — the redesign stays usable in both light and dark themes, not just dark.
- **Optional shower-detection panel and three new entity roles:** `shower_detected`, `shower_trigger_temperature`, `shower_pipe_temperature` (added to `src/types/entity-roles.ts`, `src/manufacturers/altair.ts`'s supported roles, and `src/utils/icons.ts`). Unconfigured, the System Overview panel simply expands to fill the row. Configured but inactive, a compact "No shower detected" card renders. Active (`shower_detected` is `on`), a purple panel shows an inline-SVG shower illustration with a `prefers-reduced-motion`-respecting falling-droplet animation, boost state, pipe temperature, trigger temperature, and a re-arm temperature computed as trigger − 10°C (generic UI math over the two roles, not a manufacturer conditional) — matching the `ha-altair-mvhr` integration's shower-detector rearm rule. Any individual unavailable/unmapped sensor omits its row rather than showing a placeholder or synthesized value.
- 8 new tests in `tests/unit/card-rendering.test.ts` covering the redesign: shower panel absent/inactive/active states, trigger and re-arm temperature rendering, an unavailable shower sensor not producing a fake value, boost-active and boost-remaining display, and mobile-layout structure.
- **Visual-polish follow-up to the `display_mode: system` redesign**, from first-look feedback against a live Altair install:
  - The heat-recovery percentage moved from a small pill in the System Overview panel heading into a large circular badge centred over the unit's exchanger graphic, so the unit itself — not a corner label — is what explains "how MVHR works" at a glance. The unit, its fans, and the exchanger graphic are all further enlarged, and the four duct stubs on the unit pick up the same blue (supply/outdoor) / orange (extract) colour families as the surrounding air-path panels.
  - `.mvhr-system` gained `container-type: inline-size`, and `.system-main`/`.system-lower-grid` gained `@container` rules alongside their existing `@media` ones. A plain viewport-width `@media` breakpoint can't tell that a card has been placed in a narrower dashboard column than the browser window itself allows — which is what "the whole dashboard pushed to the right" turned out to be — so the layout now also reflows based on the card's own rendered width. Purely additive: browsers without container-query support keep the previous `@media`-only behaviour.
  - The Airflow card's gauge is roughly doubled (140px → 260px, with matching larger value text) — "the single most important metric people will look at."
  - Calibration status/progress and individual fan RPM in the "More controls" drawer are now a compact 2-column tile grid (`.compact-stats-card`) instead of four full-width rows.
  - The System Status card shows coloured badges ("Boost Active", "System OK", "Filter 352 days" — tone always paired with the words, never colour alone) instead of label/value rows, and boost remaining, when active, gets its own prominent countdown callout above the badges rather than being one more small line. The header's boost pill also now shows the remaining time once boost is active.
  - The Airflow card's "Mapped level" row is now labelled "Current profile" — same role and value, just not implementation jargon.
  - 8 new tests in `tests/unit/card-rendering.test.ts` covering the container-query fallback, the relocated recovery badge (including the case where the heat-recovery calculation itself is invalid), the larger gauge, the compact stats grid, the status badges, and the boost-remaining countdown in both the status card and the header pill.
- **Second visual-polish round on `display_mode: system`**, from continued first-look review (score moved from 8.5/10 to 9.3/10 on the first round; this addresses the remainder):
  - **The shower panel moved out of the two-column layout entirely.** It's now a compact, always-on `.shower-pill` status indicator next to the boost pill in the header ("No shower detected" when idle — a read-only status, not a control) with the full illustrated purple panel (`_systemShowerBanner`) appearing as a full-width banner directly above System Overview, and only while a shower is actually active. The old `.shower-inactive` card (which sat alone in a side column even when nothing was happening) is gone, along with the whitespace it created.
  - **System Overview is the sole hero of the upper section now** — `.system-main` is a plain full-width wrapper (no more two-column grid splitting it against the shower column), and the unit/exchanger/fans/recovery badge are all enlarged again on top of the first round's sizing.
  - **Duct "particles":** the four duct stubs on the unit itself now show a small animated dot pattern travelling in the correct direction (outward on supply/exhaust, inward on extract/outdoor) while the system is active, and the air-path panels' existing flow animation switched from a translating stripe to a dot pattern too — "make the exchanger look more alive." Both are gated on `.active` and disabled under `prefers-reduced-motion: reduce`, same as every other animation in this panel.
  - The Airflow card's gauge now stacks the number and its unit on separate lines ("120" / "m³/h") instead of one run-on string, and the number is larger again.
  - **Fixed:** the System Status card's boost-remaining countdown (and the shower banner's) could show a misleading "0 min" when boost wasn't actually running, because the check only looked at the sensor having a value, not at `boost_active`. Both now require boost to actually be on.
  - The header's Mode/Boost/Shower controls sit inside a bordered, tinted "control panel" strip instead of floating loose in the corner — "feels like part of the appliance."
  - **New `calibration_start_control` entity role** (`src/types/entity-roles.ts`, `src/utils/icons.ts`) — a second action role alongside `filter_reset_control`, same fire-and-forget "press" pattern, rendered through the dedicated airflow calibration panel in system mode. Altair now declares the calibration roles supported because the backend exposes them; other manufacturer profiles can opt in with `feature_flags` when their integrations expose equivalent controls. `CONTROL_ROLES`/`_controlRow` carry per-role action/pending button text ("Run"/"Running…") instead of a single hard-coded "Reset"/"Resetting…" pair, now that there are two action roles.
  - 11 new tests in `tests/unit/card-rendering.test.ts` covering the header shower pill (idle/unavailable/absent states), the boost-remaining "0 min" fix, the stacked gauge value/unit, the full-width System Overview wrapper, the duct-particle animation and its reduced-motion coverage, the bordered header control panel, and the calibration button (including that it stays absent for a profile that hasn't opted in).
- **Micro-animations pass on `display_mode: system`** — deliberately *not* more visual complexity on the exchanger graphic itself ("I wouldn't make the exchanger itself any more complicated... instead spend time on micro-animations"):
  - **Boost speeds things up.** While boost is active, the duct particles, the air-path flow dots, and the fan spin all shorten their existing animation durations (e.g. fans 6s → 3.5s) instead of gaining any new visual element — a real boost mode raises fan speed noticeably, so the existing motion just reflects that.
  - **The heat-recovery badge gently pulses** (a single 0.7s scale-up-and-settle, not a loop) the moment the recovery percentage actually changes between two Home Assistant updates — never on first load, and never again if the next update reports the same figure.   - **The Airflow card briefly brightens** (a 0.9s border/glow flash) the moment the current-airflow reading increases from the previous update — never on first load, and never on a decrease. Tracked the same way via `_prevAirflowNumber`.

  - 5 new tests in `tests/unit/card-rendering.test.ts` covering the boost animation-duration overrides and the one-shot change-detection for both the recovery pulse and the airflow brighten (first render, an increase/change, and a subsequent unchanged/decreased render).
- **The Airflow gauge's arc now reflects the configured operating level, not "current airflow ÷ target airflow."** `mapped_level` (Altair's 0-10 speed scale, read directly as 0-100%) is the preferred source; a new **`selected_speed` entity role** (`src/types/entity-roles.ts`, `src/utils/icons.ts`) — same 0-10 concept, a different entity — is the fallback when `mapped_level` isn't available. Not declared supported by any manufacturer profile by default (same opt-in pattern as `filter_reset_control`/`calibration_start_control`: `feature_flags: { selected_speed: true }`), since no manufacturer doc confirms it as a capability distinct from `mapped_level` yet. The gauge's big central number is unchanged — still the actual measured airflow in m³/h — and target airflow remains a separate detail row, never the gauge's maximum.
  - 6 new tests in `tests/unit/card-rendering.test.ts` covering the arc fraction at `mapped_level` 0/4/6/10, confirming current/target airflow no longer influence the fraction, the `selected_speed` fallback when `mapped_level` is unavailable, and an empty arc when neither is available.
- **The system-mode MVHR unit now contains a functional four-path SVG airflow schematic.** Extract and outdoor air curve inward through separate warm/cool exchanger passages; exhaust and supply curve outward without the streams visually merging. Live temperature gradients, particles, boost animation, unavailable-state muting, theme support, and the outdoor-left/indoor-right physical layout remain reactive overlays.

### Changed

- **Rebuilt the `display_mode: detailed` layout as a single unified MVHR dashboard.** Previously the legacy compact/diagnostic content (`Temperatures`/`Airflow`/`System status` sections, added in Phase 2) rendered unconditionally *above* the Phase-3 dashboard in every display mode — on a live card this looked like the old compact card with a small dashboard bolted on underneath. `display_mode: detailed` now renders the dashboard alone: a large airflow visual (four labelled air paths around a bigger MVHR unit illustration, with a heat-recovery badge inside it), a restyled controls panel (mode/boost/override, 44px+ touch targets, active-mode highlighting), a metrics tile grid, and a single status strip. `display_mode: homeowner` is unchanged — it still renders the original compact content.
- Header (both display modes): title, a status dot, and the current mode now read together on one line, with the subtitle below (`Altair MVHR ● Home` / `Heat Recovery Ventilation System`). The header's availability indicator now only reflects required-entity failures — an unmapped or unavailable optional entity (fault, frost protection, boost/override controls, calibration metadata) no longer triggers a "1 sensor unavailable"-style warning at the top level (`src/data/availability-summary.ts` gained an `ignoreRoles` option for this).
- Dashboard layout uses CSS grid with `fr`/`minmax()` throughout — no fixed pixel widths — at three breakpoints: desktop (~70/30 visual/controls split), tablet (≤900px, single column, metrics in a denser auto-fit grid), and mobile (≤599px, compact 2×2 air-path grid around the unit, metrics locked to 2 columns, no horizontal scroll).
- The bottom status strip now reports one of "System OK", "Communication issue" (a required, configured entity is unavailable/missing), "Fault detected", "Calibrating…", or "Calibration required", instead of duplicating the filter/calibration text that's already shown in the metrics tiles.

- Project foundation: `docs/architecture.md`, `SPECIFICATION.md`, `README.md`, `ROADMAP.md`, `CONTRIBUTING.md`, `CLAUDE.md`.
- Repository structure finalized (`docs/architecture.md` §2), including `src/data/`, `src/editor/`, `tests/fixtures/`, `docs/manufacturers/`, and `examples/generic/`.
- Tooling scaffolded: TypeScript, ESLint, Prettier, Vitest, GitHub Actions (`build.yml`, `release.yml`), HACS metadata (`hacs.json`), `package.json`.
- Core data layer: entity role registry, `CapabilityProfile`/`MvhrSnapshot`/`CardConfig` types, config schema validation, capability resolver, entity resolver.
- Manufacturer capability profiles: Altair 160, Zehnder ComfoAir Q (Q350/Q450/Q600), Aerofresh (300/450), Generic — with test fixtures for each.
- Working `custom:hiper-mvhr-card` registration (guarded against duplicate registration on dev-server hot reload) and a minimal rendering vertical slice.
- Local dev preview (`npm run dev` → `dev/preview.html`) rendering the card against realistic mock Home Assistant states, no live HA instance required.
- **Phase 2 card layout** (merged to `main`): header (name, manufacturer/model, operating mode, overall availability indicator), a responsive temperature grid (outdoor/supply/extract/exhaust), an airflow section, and a system status section (summer bypass, filter, fault, frost protection).
- Three new entity roles: `filter_remaining`, `fault_active`, `frost_protection_active`.
- A fifth rendering state, `entity_missing`, distinguishing a misconfigured/renamed entity (a configuration problem) from a real sensor that's currently `unavailable`/`unknown` (a runtime state) — previously conflated in Phase 1.
- `display_mode: homeowner | detailed` — homeowner omits unconfigured optional roles and raw entity IDs; detailed shows "not configured" rows and explicit missing-entity warnings naming the entity id.
- Accessibility: heading hierarchy, `role="status"` on the availability indicator, tone always paired with an icon/word (never color alone), `aria-label`s on each section.
- Icon support via Home Assistant's own `<ha-icon>` (Material Design Icons), no icon package bundled.
- **Phase 3A — first interactive control** (`filter_reset_control`): a fire-and-forget "press" action, rendered as a native, labeled, keyboard-operable `<button>` in the system status section. Enabled today only via the `generic` profile's `feature_flags` — Altair/Zehnder/Aerofresh don't declare it supported by default, since filter resettability is still TBD for all three (`SPECIFICATION.md` §3).
- `src/data/control-dispatcher.ts`: a generic, reusable action-dispatch mechanism (idle/pending/error state, timeout guard) that calls the Home Assistant service convention for "press" actions (`button.press`/`input_button.press`, derived from the entity id's domain — not manufacturer-specific). No optimistic target-value reconciliation yet — that lands with Phase 3B's mode selector, which has a value to reconcile against; filter reset doesn't.
- `hass.callService` remains optional on the `HomeAssistant` type; the dispatcher and component no-op safely when it's absent (dev preview, minimal test fixtures) rather than requiring it.

### Changed

- **Breaking (pre-1.0):** the Aerofresh capability profile's id changed from `aerfresh` to `vent_axia_sentinel_econiq`, reflecting that it's the Vent-Axia Sentinel Econiq control platform sold under the Aerofresh brand — existing `manufacturer: aerfresh` configs need updating to `manufacturer: vent_axia_sentinel_econiq`. The user-facing brand name shown on the card is unchanged: always "Aerofresh."
- **Breaking (pre-1.0):** card config field `view` renamed to `display_mode`; accepted values narrowed from `homeowner | installer | commissioning` to `homeowner | detailed` (only the two implemented so far — `commissioning` will be added back once it's built, ROADMAP.md Phase 4).
- Corrected a spec/implementation mismatch: `SPECIFICATION.md` previously documented a `title` config field that was never implemented; the actual field has always been `name`. The spec now matches the code.

### Fixed

- `filter_reset_control` never reached its interactive "ok" state for a real, never-pressed `button`/`input_button` entity: `entity-resolver.ts` treated every `unknown` state as `unavailable`, but `unknown` is that domain's normal idle state before a first press, not a runtime problem. Fixed with a domain-aware exemption (a documented fact about the `button`/`input_button` HA platforms generically, not a manufacturer-specific check) — `unavailable` itself is still treated as unavailable for these entities.
- `feature_flags` values were coerced with `Boolean(enabled)`, so the truthy string `"false"` silently enabled a flag. `parseConfig` now requires a literal boolean and throws `ConfigValidationError` otherwise.
- Homeowner mode's header availability chip could still show the configuration-jargon fallback label "Not configured" when nothing had been mapped yet, contradicting homeowner mode's "no configuration jargon" rule. The chip is now omitted in that specific case in homeowner mode; detailed mode is unaffected.
- `vite.config.ts` rewritten to be valid native ESM (`import.meta.url` + `fileURLToPath` instead of a nonexistent CommonJS `__dirname`).
- A card-rendering test asserted on raw `shadowRoot.textContent` for the "nothing rendered yet" case, which broke under happy-dom's non-spec-compliant handling of Lit's internal empty-template marker node. Rewritten to check for the absence of an `ha-card` element and of any `Text` node content specifically.
- Upgraded `@typescript-eslint/*` to v8 (was v7, incompatible with TypeScript 5.9+), `vite` to ^6.2.0 and `vitest` to ^4.1.0 (both peer-matched), and `happy-dom` to ^20.0.0 (was ^14.0.0, vulnerable to a VM-context-escape RCE, GHSA-37j7-fg3j-429f — a test-only dependency, never shipped in the bundle). Added `@types/node`.
